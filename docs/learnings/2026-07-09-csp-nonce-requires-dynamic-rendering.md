# Dashboard (and almost every page) dead in production ‚Äî nonce CSP vs static prerender

**Problem** ‚Äî "Dashboard won't load" on the deployed app: browser console full of `Executing inline script violates the following Content Security Policy directive 'script-src ‚Ä¶ 'nonce-‚Ä¶''` errors; page renders server HTML but never hydrates.

**Approach** ‚Äî (1) Filtered the console paste: `content.js` / `chrome-extension://` lines are extension noise; the `dashboard:1` CSP violations are the app. (2) Compared the CSP header nonce against `nonce="‚Ä¶"` attributes in the HTML **of the same response** (curl -D header dump + body to files) ‚Äî comparing across two separate requests is meaningless because the nonce rotates per request. (3) `/login`: header nonce present on 23/23 script tags ‚Üí works. `/dashboard`: header has a nonce, HTML has **zero** nonce attributes, `x-vercel-cache: HIT` ‚Üí static prerender served from cache. (4) Probed every route: all pages except `/login` (which exports `force-dynamic`) were `PRERENDER`/`HIT` with no nonces ‚Äî the whole app was dead, dashboard was just where the user looked. (5) Vercel build-log route table (`∆í` vs `‚óã`) confirmed and later verified the fix without needing to reproduce locally.

**Solution** ‚Äî `export const dynamic = 'force-dynamic'` in `src/app/layout.tsx` (root layout, cascades to every page). Nonce-based CSP fundamentally requires per-request rendering: prerendered HTML is baked at build time when no request nonce exists. Deployed via `vercel deploy` (preview, build verified from its route table since preview URLs sit behind Vercel SSO) then `vercel deploy --prod`. Verified on production: header nonce matches 100% of script tags on `/`, `/login`, `/dashboard`, `/calendar`, `/budget`.

**Rule** ‚Äî If a middleware mints a per-request nonce CSP, every HTML route must render dynamically; a single static/ISR page under that middleware ships nonce-less inline scripts that the browser will block, and the page will show HTML but never hydrate. When diagnosing CSP nonce failures, always compare header vs HTML nonce within one response, and read `x-vercel-cache` ‚Äî `HIT`/`PRERENDER` on a nonce'd page is the whole diagnosis. Local dev can't reproduce this (dev renders everything dynamically); only a production build shows it.

**Dead ends** ‚Äî Blaming the Chrome extension errors mixed into the console output. Comparing header nonce from one request with HTML from another (false mismatch on /login sent me toward a caching theory that `x-vercel-cache: MISS` then disproved). Trying to verify on the preview URL's pages ‚Äî Vercel deployment protection 302s everything; the build log's route table is the reliable preview-side check.
